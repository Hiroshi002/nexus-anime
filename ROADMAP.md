# Roadmap

This document is the public-facing counterpart to the internal milestone table in `docs/master-roadmap.md`. It is organized as a phased delivery plan from where the project is today to a stable **v1.0.0** public release.

Each milestone below is a **release boundary** — it ships a coherent, tested increment of product capability. Milestones map to versions as described in [Repository Design §9](docs/REPOSITORY-DESIGN.md) and the [Versioning](#versioning) section at the bottom.

---

## Current state

- **v0.2.x** is the active release line (pre-release).
- Milestones M0 (Repository scaffold), M1 (Design system), and M2 (Catalog foundation — DB, cache, API envelope, error boundaries) are complete on `main`.
- M3 (Auth) is in progress.

---

## Milestones

### M3 — Auth complete → `v0.3.0`

- Auth.js v5 integration with sessionStrategy.
- Credential login (email + password) and session management.
- OAuth providers (Google, GitHub) with callback handling.
- Auth-protected API route guards and `<SessionProvider>` wiring.
- Middleware to protect app routes and redirect unauthenticated users.

**Exit criterion:** users can sign in, sign out, and stay signed in across restarts without errors.

---

### M4 — User profiles, watchlist, continue-watching → `v0.4.0`

- User profile page (avatar, display name, preferences).
- Watchlist: add / remove anime, persisted via `@nexus/db`.
- Continue-watching: resume from last-played position per episode.
- Server Actions for data mutations; optimistic UI updates on the client.

**Exit criterion:** a signed-in user can build a watchlist and resume playback within a single session and after reloading.

---

### M5 — Payments → `v0.5.0`

- Stripe integration: products, prices, and subscription lifecycle.
- Webhook handler for `checkout.session.completed`, `customer.subscription.updated`, etc.
- Customer portal link for self-service plan management.
- Entitlement gating (free vs premium tiers) middleware and UI states.

**Exit criterion:** a user can upgrade to a premium plan, access gated content, and cancel without leaving the app.

---

### M6 — Video streaming → `v0.6.0` – `v0.9.0`

Iterative delivery (may span multiple point releases):

- Cloudflare Stream ingestion and playback token flow.
- Adaptive bitrate HLS player with custom controls (cinematic UI).
- Episode → video asset mapping; progress tracking.
- Subtitle track support and audio track selection.
- Offline download handling (later sub-milestone, if applicable).

**Exit criterion:** premium users can play any episode from the catalog with progress sync across devices.

---

### M7 — Public launch → `v1.0.0`

- Performance: Core Web Vitals targets met on key pages (LCP < 2.5s, INP < 200ms, CLS < 0.1).
- SEO: metadata, Open Graph images, sitemap, robots.txt.
- Analytics and error monitoring (Sentry or comparable).
- Accessibility: WCAG 2.1 AA pass on primary user flows.
- On-call runbook and production deployment checklist.
- Public documentation site (if applicable).

**Exit criterion:** the product is feature-complete, stable, and ready for an uncontrolled public audience. `v1.0.0` is tagged.

---

## Pre-launch hardening (`v0.9.x`)

Before M7, a dedicated hardening cycle:

- Load testing on catalog and video endpoints.
- Security review of auth, payments, and video token flows.
- Rate limiting and abuse protections via `@nexus/cache`.
- Graceful degradation paths for all third-party failures (Stripe, Cloudflare, Upstash, Neon).
- Data backup and recovery drill.

## Versioning

- **M0–M2** → `v0.2.x` (pre-release, shipped).
- **M3** → `v0.3.0`.
- **M4** → `v0.4.0`.
- **M5** → `v0.5.0`.
- **M6** → `v0.6.0` – `v0.9.x`.
- Pre-launch hardening → `v0.9.x`.
- **M7** → `v1.0.0`.

Pre-release suffixes (`-alpha.N`, `-beta.N`, `-rc.N`) are used for any release that is not yet considered stable. See [Repository Design §9](docs/REPOSITORY-DESIGN.md) for the full strategy.

- GitHub Milestones mirror the list above and group related issues/PRs.

- Each milestone has a detailed spec in `docs/milestones/milestone-<N>-<title>.md`.

## Contributing to the roadmap

The roadmap is a living document. To propose a change:

1. Open an issue using the Feature Request template and tag it `scope:docs`.
2. Explain the rationale and which milestone it affects.
3. A maintainer will discuss and, if accepted, open a PR against this file.

Do not edit this file directly without discussion.
