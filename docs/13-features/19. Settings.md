# Settings — Nexus Anime

> **Audience:** Engineers, Product, Design, Legal (GDPR)
> **Milestone:** M4 (Account, Preferences, Privacy, Notifications); M5 (Connected accounts); M7 (full data portability)
> **Owner:** Engineering
> **Status:** Draft

---

## 1. Purpose

Settings is the authenticated user's control center for identity, preferences, privacy, notifications, and connected accounts. It exists to give users agency over their experience (language, playback, subtitle style), their privacy (visibility, data export, account deletion), and their integrations (OAuth links). It is the primary surface for GDPR/CCPA obligations (data export, account deletion) and for account security (password change, email change, 2FA).

## 2. Business Goals

- **Trust/compliance** — meet GDPR data-export and account-deletion requirements; target zero compliance incidents related to data subject requests.
- **Retention** — let users tailor playback, language, and notifications to their taste; target 6% lift in 30-day return rate for users who customize ≥2 preferences.
- **Security** — self-serve password change and account deletion reduce support load and limit the window of compromised credentials.
- **Conversion** — connected-accounts management (link/unlink OAuth) reduces login friction and supports premium onboarding.

## 3. Functional Requirements

### 3.1 Happy Path

1. A logged-in user navigates to `/settings` and sees sections: Account, Preferences, Privacy, Notifications, Connected accounts.
2. In Account, the user changes their email (with verification), changes their password (current + new), or initiates account deletion (with confirmation + grace period).
3. In Preferences, the user sets language, default playback quality, autoplay behavior, and subtitle style (font/color/size/background).
4. In Privacy, the user toggles profile and watchlist visibility, requests a GDPR data export, or downloads their data.
5. In Notifications, the user toggles per-type email/push/in-app preferences.
6. In Connected accounts, the user links or unlinks Google/GitHub OAuth providers.

### 3.2 Alternate Flows

1. **Email change** — user submits new email; system sends a verification link to the new address; the change only takes effect on confirmation. The old email is notified of the pending change.
2. **Account deletion with active subscription** — system blocks deletion until the Stripe subscription is canceled; user is redirected to cancel first.
3. **Data export for long-history user** — export is async; user receives an in-app notification + email when the archive is ready for download (≤24h).
4. **Unlink last auth method** — system prevents unlinking the last provider if no password is set (would lock the user out); prompts to set a password first.
5. **Settings for deleted OAuth provider** — if a provider is deprecated, the Connected accounts section shows it as "Unavailable" with an unlink-only option.

### 3.3 Edge Cases

1. Change email to an existing email → `EMAIL_TAKEN` (409); form shows inline error.
2. Delete account with active subscription → blocked with a clear message and link to cancel subscription.
3. Data export for user with 10 years of history → async job; capped at 24h; archive split into paginated files if >100 MB.
4. Unlink last auth method without password → blocked; prompt to set password first.
5. Settings for deleted OAuth provider → shown as "Unavailable"; unlink-only.
6. Concurrent settings edits → optimistic concurrency via `users.version`; conflict returns `CONFLICT` (409) with a refresh prompt.

## 4. Non-Functional Requirements

- **Performance:** Settings page p95 < 250ms (SSR); preference PATCH p95 < 200ms; data export enqueues < 500ms.
- **Availability:** 99.9% read path; write path tolerates brief degradation; account deletion is idempotent and recoverable within a grace period.
- **Scalability:** 500 settings reads/sec; 50 preference writes/sec; data export is queue-backed and does not block the request path.
- **Accessibility:** Each section is a labeled form; keyboard-navigable; destructive actions require confirmation and are undoable within the grace period. WCAG 2.1 AA.
- **Localization:** All settings labels externalized; language preference drives UI locale; future RTL via logical properties.
- **Security:** All endpoints authenticated; email change requires verification; account deletion requires password/re-auth + confirmation token; 2FA (when enabled) required for sensitive changes; all sensitive actions logged to `audit_log`.

## 5. User Stories

- As a **logged-in user**, I want to change my email and password so that my account stays secure and up to date.
- As a **logged-in user**, I want to set my language, playback, and subtitle preferences so that the app feels like mine.
- As a **logged-in user**, I want to control my profile and watchlist visibility so that I decide who sees my activity.
- As a **logged-in user**, I want to export or delete my data so that I'm in control of my information.
- As a **logged-in user**, I want to link/unlink Google and GitHub so that I can sign in however I prefer.
- As a **legal/compliance officer**, I want data-export and account-deletion flows so that we meet GDPR/CCPA obligations.

## 6. Acceptance Criteria

- [ ] Changing email sends a verification link to the new address; the change only applies on confirmation; the old email is notified.
- [ ] Changing password requires the current password; all other sessions are invalidated; user stays signed in on the current device.
- [ ] Account deletion requires password/re-auth + a typed confirmation; a 14-day grace period allows cancellation; after grace period, data is purged per retention policy.
- [ ] Account deletion is blocked while an active Stripe subscription exists, with a clear message and link to cancel.
- [ ] Data export enqueues an async job; user is notified when ready; archive is available for 7 days; export completes within 24h even for long-history users.
- [ ] Unlinking the last OAuth provider is blocked if no password is set; user is prompted to set a password first.
- [ ] Per-type notification preferences persist and are respected by the notification creation pipeline.
- [ ] All sensitive actions (email/password/delete/unlink) are written immutably to `audit_log`.

## 10. Edge Cases

1. **Change email to existing email** — UNIQUE constraint on `users.email` (citent); returns `EMAIL_TAKEN` (409); form shows inline error; no verification sent.
2. **Delete account with active subscription** — service checks Stripe; if active, returns `SUBSCRIPTION_ACTIVE` (409) with a link to cancel; deletion is blocked until canceled.
3. **Data export for long-history user** — async job paginates through history; if archive >100 MB, splits into multiple files; capped at 24h; user notified on completion; archive expires after 7 days.
4. **Unlink last auth method** — service checks remaining providers + password hash; if none, returns `CANNOT_UNLINK_LAST_AUTH` (409) and prompts to set a password first.
5. **Settings for deleted OAuth provider** — provider marked `deprecated` in config; UI shows "Unavailable" with unlink-only; no new links allowed.
6. **Concurrent settings edits** — `users.version` checked on UPDATE; mismatch returns `CONFLICT` (409); client refreshes and re-applies.
7. **Account deletion grace period** — `deletion_scheduled_at` set to now + 14 days; login during grace offers "Cancel deletion"; after grace, a job purges PII and soft-deletes the row.
8. **Email change while another session pending** — only one `pending_email` at a time; new request replaces pending and re-sends verification.

## 13. Security Considerations

- **OWASP A01 (Broken Access Control)** — all endpoints authenticated; users can only modify their own settings; ownership re-checked on every mutation.
- **OWASP A02 (Cryptographic failures)** — passwords hashed with bcrypt (cost 12); current password required for sensitive changes; 2FA (when enabled) required for email/password/delete.
- **OWASP A03 (Injection)** — Zod-validated inputs; subtitle style values whitelisted; no raw HTML in any preference.
- **Account deletion** — requires re-auth + typed confirmation; 14-day grace period; idempotent; all sessions invalidated on confirmation.
- **Email change** — verification on the new address; old email notified; pending change visible in settings.
- **Audit** — every sensitive action (password, email, delete, unlink) written immutably to `audit_log` with actor, action, timestamp, and IP hash.
- **PII** — data export is scoped to the requesting user only; archives are signed, expiring, and access-logged.

## 14. Performance Requirements

- Settings page p95 < 250ms (SSR, always fresh — no stale preferences).
- Preference PATCH p95 < 200ms; write-through invalidates any cached preference reads.
- Password/email change p95 < 400ms (bcrypt cost).
- Data export enqueue p95 < 500ms; async job completes within 24h.
- Rendering strategy: settings shell is SSR; each section is a Suspense boundary with skeleton; forms are client islands.
- Client bundle budget for settings islands: < 20 kB gzipped total.
