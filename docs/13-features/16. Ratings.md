# Ratings — Nexus Anime

> **Audience:** Engineers, Product, Design
> **Milestone:** M4
> **Owner:** Product
> **Status:** Draft

---

## 1. Purpose

The Ratings feature lets authenticated users score anime on a 1–10 integer scale and surfaces an aggregate score (average, distribution, count) on every anime detail page. It exists to give the catalog a trust signal for discovery, to anchor the recommendation engine, and to give returning users a visible record of their taste. Aggregates are public; the act of rating is authenticated to preserve one-person-one-vote integrity.

## 2. Business Goals

- **Engagement** — increase catalog dwell time by surfacing a social proof signal on detail pages; target 15% lift in detail→play conversion within 60 days of launch.
- **Retention** — give returning users a "my ratings" touchpoint in their profile; target 5% lift in 30-day return rate for users who have rated ≥5 titles.
- **Data quality** — power collaborative-filtering recommendations; target measurable improvement (10%+) in recommendation click-through once ratings signal is available.
- **Trust** — transparent, tamper-resistant aggregates (verified-watch gating, rate limits) reduce review-bomb perception and support premium brand positioning.

## 3. Functional Requirements

### 3.1 Happy Path

1. A logged-in user opens an anime detail page and sees the current aggregate rating (average, count, distribution bar).
2. The user taps a star/number control, selects a value 1–10, and submits; the UI optimistically reflects their rating and the aggregate updates.
3. The user returns to the same detail page later and sees their existing rating pre-selected; they can change it or remove it.
4. An unauthenticated visitor views the same detail page and sees the aggregate but no rating control (prompts sign-in on tap).
5. The profile "My Ratings" tab lists every title the user has rated, filterable and sortable.

### 3.2 Alternate Flows

1. **Update path** — user re-rates an already-rated anime: system performs an UPDATE (not a second row), preserving the original `created_at` and setting `updated_at` to now.
2. **Verified-watch waiver** — user rates a title they have not watched; rating is accepted but flagged `verified = false` and excluded from the verified-only aggregate toggle.
3. **Anonymous deep link** — unauthenticated user follows a link to a rating-deep-linked URL, sees aggregate + a "Sign in to rate" prompt; after auth, returns to the detail page with the rating control focused.
4. **Empty state** — anime with 0 ratings shows "No ratings yet — be the first" instead of a numeric average.
5. **Hover preview** — hovering the control on desktop shows a tooltip mapping 1–10 to a label (e.g. 9–10 = "Masterpiece", 1–2 = "Awful").

### 3.3 Edge Cases

1. User rates the same anime twice in rapid succession (double-click / retry) → idempotent UPDATE; no duplicate row.
2. User rates an anime that has since been soft-deleted → `ANIME_NOT_FOUND` (404).
3. User submits a non-integer or out-of-range value → `VALIDATION_ERROR` from Zod (`too_small`/`too_big`).
4. User removes the only rating on a title → `average_rating` and `rating_count` revert to `NULL`/`0`; empty-state UI appears.
5. Rating on a title not in local catalog (new upstream ID) → 404 with clear message; ratings are only allowed on cataloged anime.
6. Mass rating from one account in short window → rate-limited (see §11/§13); subsequent requests return `RATE_LIMITED` with `retryAfter`.

## 4. Non-Functional Requirements

- **Performance:** Aggregate read p95 < 100ms (cached); rate/submit p95 < 250ms; distribution computation p95 < 150ms.
- **Availability:** 99.9% for read path; write path tolerates brief degradation (user can retry).
- **Scalability:** Support 50 rating writes/sec burst at launch; aggregate reads at 500 req/s sustained via Redis cache (`nexus:anime:{id}:rating`).
- **Accessibility:** Rating control is a radiogroup of 10 buttons; keyboard-navigable; labels exposed to screen readers; focus ring visible. WCAG 2.1 AA.
- **Localization:** Rating UI string externalized; numeric scale is culturally neutral (no conversion needed); tooltip verbs localized.
- **Security:** Authenticated-only writes; anti-gaming via verified-watch gate (optional) + rate limits; no PII leaked in aggregate endpoints; immune to mass-via-script via CSRF double-submit.

## 5. User Stories

- As a **logged-in user**, I want to rate an anime I just finished so that my taste is recorded and others can discover it.
- As a **logged-in user**, I want to change or remove a rating so that my score stays accurate.
- As a **visitor**, I want to see the average rating and distribution on a detail page so that I can decide whether to watch.
- As a **returning user**, I want a list of everything I've rated so that I can remember what I thought and revisit favorites.
- As a **product manager**, I want rate-limited, verified-gated ratings so that aggregates stay trustworthy.

## 6. Acceptance Criteria

- [ ] Aggregating endpoint returns `{ average: number|null, count: number, distribution: number[10] }` and matches DB recomputation within rounding tolerance.
- [ ] Submitting a rating by an authenticated, cataloged-anime case inserts exactly one row; resubmitting updates in place (row count unchanged).
- [ ] Unauthenticated POST/PUT/DELETE to any rating endpoint returns `UNAUTHORIZED` (401) without modifying state.
- [ ] An anime with 0 ratings returns `average: null, count: 0` and renders the empty-state UI, not `0.0`.
- [ ] Exceeding 20 rating writes in 60 seconds from one user returns `RATE_LIMITED` (429) with a `retryAfter` header; legitimate traffic is unaffected.
- [ ] A rating on a soft-deleted anime returns `ANIME_NOT_FOUND` (404).
- [ ] Distribution array sums to `count`; bucket indices 0–9 map to scores 1–10.
- [ ] Rating control is reachable and operable by keyboard; each option has an accessible name.
- [ ] Aggregate cache invalidates within 5 seconds of a successful write.

## 9. Database Dependencies

| Table / View | Operation                         | Index / Query Notes                                                                                            |
| ------------ | --------------------------------- | -------------------------------------------------------------------------------------------------------------- |
| `ratings`    | INSERT / UPDATE / DELETE / SELECT | PK `id`; UNIQUE `(user_id, anime_id)` enforces one-per-user; indexes on `anime_id`, `user_id`                  |
| `anime`      | SELECT / UPDATE                   | Aggregate columns `average_rating numeric(3,2)`, `rating_count int`; `idx_anime_rating_count` for leaderboards |

Zod schemas:

- `RatingValueSchema: z.number().int().min(1).max(10)`.
- `CreateRatingSchema: z.object({ animeId: z.string().uuid(), value: RatingValueSchema })`.
- `rating_status` defaults `verified: boolean` derived from watch history existence at write time.

## 10. Edge Cases

1. **Duplicate rating (race condition)** — two concurrent POSTs for the same `(user_id, anime_id)`: DB unique constraint wins; second request returns `DUPLICATE_RATING` (409) and client falls back to PUT.
2. **Anime with no ratings** — aggregate endpoint returns `average_rating: null`; UI renders empty-state copy and a zero-height (not absent) distribution so layout doesn't shift.
3. **Rating unwatched title** — accepted with `verified = false`; visible to user; included in raw aggregate but excluded from the (future) verified-only toggle. User sees a subtle "unverified" indicator.
4. **Removing the last rating** — DELETE decrements count and recomputes average; at zero, `average_rating` reverts to `NULL`. Cache invalidated.
5. **Deleted anime** — FK with `ON DELETE CASCADE`; deleting anime removes its ratings. Reads against deleted anime return `ANIME_NOT_FOUND`.
6. **Out-of-range payload** — Zod rejects before DB; returns `VALIDATION_ERROR` with `details.issues` on the `value` path.
7. **Burst writes (anti-bot)** — sliding-window rate limit of 20 writes/min/user; excess is `RATE_LIMITED`. Sustained abuse flags account for review (see §13).
8. **Re-rating via PUT** — updates `value` and `updated_at`; preserves `created_at` and original `verified` flag unless watch state changed since first rating.

## 11. Error Handling

| Error Condition                | User-Facing Message                              | Recovery Action                                    | Log Level |
| ------------------------------ | ------------------------------------------------ | -------------------------------------------------- | --------- |
| Unauthenticated write          | "Sign in to rate this title."                    | Redirect to `/auth/signin?returnTo=`               | info      |
| Out-of-range value             | "Rating must be between 1 and 10."               | Keep invalid field focused; show inline            | warn      |
| Duplicate rating (409 on POST) | "You've already rated this."                     | Silent fallback to PUT/update                      | warn      |
| Anime not found                | "This anime is no longer available."             | Return to catalog                                  | warn      |
| Rate limit exceeded            | "You're doing that too fast. Try again shortly." | Retry after `retryAfter` seconds                   | warn      |
| Upstream/catalog unavailable   | "Ratings are temporarily unavailable."           | Retry with backoff; show stale aggregate if cached | error     |
| Server error                   | "Something went wrong. Please try again."        | Generic retry                                      | error     |

## 13. Security Considerations

- **OWASP A01 (Broken Access Control)** — all writes require valid session; ownership checked on DELETE/PUT (user can only modify their own ratings).
- **OWASP A04 (Insecure Design) — anti-gaming**: UNIQUE `(user_id, anime_id)`; 20 writes/min per-user rate limit (Redis sliding window); optional verified-watch gate; anomaly detection flags accounts with abnormal rating velocity for review.
- **OWASP A03 (Injection)** — value validated by Zod before DB; parameterized Drizzle queries.
- **Enumeration risk** — aggregate endpoint does not expose which user rated what; `/users/{username}/ratings` exposes only the public profile's own list.
- **PII** — no email/name in rating responses; analytics use `user_id_hash`.
- **CSRF** — all mutations required the double-submit token.
- **Mass manipulation** — per-IP and per-account write caps; suspicious bulk writes logged and quarantined from aggregate until reviewed.

## 14. Performance Requirements

- Aggregate read p95 < 100ms at 500 req/s (Redis cache warm; cache key `nexus:anime:{id}:rating`, TTL 60s; write-through invalidation on PUT/POST/DELETE).
- Write path (POST/PUT) p95 < 250ms including aggregate recompute and cache invalidation.
- Distribution query backed by a materialized count per bucket on the `anime` row to avoid `GROUP BY` on the hot path.
- Rendering strategy: Server Component for aggregate (ISR, revalidate 60s); `RatingControl` is a client island lazy-loaded under 8 kB.
- Client bundle budget for rating island: < 12 kB gzipped.
