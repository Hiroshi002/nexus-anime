# Continue Watching — Nexus Anime

> **Audience:** Engineers (frontend + backend), Product, Design
> **Milestone:** M3
> **Owner:** Engineering
> **Status:** Draft

---

## 1. Purpose

Continue Watching lets authenticated users resume playback from their last-watched position. It powers a home-page rail of in-progress anime and a dedicated `/continue-watching` page with full progress visibility. The feature reads from and writes to the `continue_watching` table, which acts as a materialized cursor derived from `watch_history`.

## 2. Business Goals

- **Increase session starts** — one-tap resume reduces friction; target +15% lift in "play" clicks from the home page.
- **Improve retention** — users who can pick up where they left off are more likely to complete a series; target +10% episode-completion rate.
- **Reduce churn** — users who finish series subscribe longer; continue-watching is the on-ramp to binge completion.
- **Cross-device continuity** — cursor sync across web, mobile, and TV keeps users inside the ecosystem.

## 3. Functional Requirements

### 3.1 Happy Path

1. User logs in and lands on the home page.
2. System queries `continue_watching` for the authenticated user, ordered by `updated_at DESC`, filtered to `progress_pct > 5 AND progress_pct < 95`.
3. A horizontal rail renders each entry with: poster, anime title, episode number, progress bar (`progress_pct`), and "Resume" button.
4. Tapping "Resume" navigates to `/watch/{animeSlug}/{episodeNumber}?t={position_seconds}`.
5. The video player seeks to `position_seconds` on mount.
6. During playback, the player sends a heartbeat (PATCH) every 30 seconds to update the cursor.
7. A dedicated `/continue-watching` page shows the same list in a larger layout with a "Remove" option per entry.

### 3.2 Alternate Flows

1. **First-time user (no history):** Rail is hidden; no entry appears. The system shows a "Start Watching" prompt instead.
2. **All episodes completed:** Entry drops out of the default rail (filtered by `progress_pct < 95`). A "Completed" filter on the dedicated page still shows it.
3. **Episode deleted:** If the cursor's `episode_id` references a soft-deleted episode, the system advances the cursor to the next available episode or removes the entry if no episodes remain.
4. **Multiple entries for same anime (different seasons):** The `unique (user_id, anime_id)` constraint means only one cursor per anime. The cursor points to the most recently watched episode across all seasons.
5. **Concurrent devices:** Two devices heartbeating simultaneously use optimistic concurrency (`version`). The losing device receives `409 Conflict` and re-reads before retrying.

### 3.3 Edge Cases

1. User with zero watch history sees an empty rail (no error).
2. User who completed every episode in their list sees an empty "In Progress" section but can view "Completed" tab.
3. Episode referenced by a cursor is soft-deleted — system advances cursor or removes the row.
4. Network failure during heartbeat — client retries with exponential backoff; stale cursor is acceptable for up to 5 minutes.
5. User watches the same episode on two tabs simultaneously — last-write-wins via `version` check.
6. Rapid seek events flood heartbeats — client debounces to one PATCH per 30 seconds.
7. Cursor points to an anime that has been fully removed from the catalog — entry is hidden from the rail; no 500.

## 4. Non-Functional Requirements

- **Performance:** Heartbeat PATCH p95 < 100ms; home-page rail query p95 < 200ms.
- **Availability:** 99.9% for read endpoints; 99.9% for heartbeat writes (best-effort; stale data is acceptable for a few seconds).
- **Scalability:** 500k concurrent users actively heartbeating at 30-second intervals (~17k writes/sec peak).
- **Accessibility:** Rail is keyboard-navigable (arrow keys); progress bars have `aria-valuenow`; screen readers announce "Resume Episode X of Y at Z%."
- **Localization:** Strings externalized; number formatting for progress percentage uses `Intl.NumberFormat`.
- **Security:** All endpoints require authentication; users may only read/write their own cursors; rate limit 30 heartbeats per 60 seconds per user.

## 5. User Stories

- As a **logged-in user**, I want to see my in-progress anime on the home page so that I can resume watching without searching.
- As a **logged-in user**, I want to pick up exactly where I left off so that I don't have to scrub through an episode.
- As a **returning user**, I want my progress synced across devices so that I can switch from my phone to my TV seamlessly.
- As a **logged-in user**, I want to remove an anime from my continue-watching list so that shows I abandoned don't clutter my rail.
- As a **logged-in user**, I want to see a progress bar per episode so that I know how much is left.

## 6. Acceptance Criteria

- [ ] Authenticated user with in-progress anime sees a "Continue Watching" rail on the home page.
- [ ] Tapping "Resume" navigates to the correct episode and seeks to the saved position.
- [ ] Heartbeat is sent every 30 seconds during playback and updates `position_seconds` and `progress_pct`.
- [ ] User with no watch history does not see the rail (no empty-state clutter on home).
- [ ] User who completed all episodes (progress_pct >= 95) does not see the entry in the default rail.
- [ ] DELETE endpoint removes the entry; subsequent GET does not return it.
- [ ] Heartbeat on a non-existent cursor returns `404`; the client must POST first.
- [ ] Concurrent heartbeat with stale `version` returns `409` with the current server state.
- [ ] `/continue-watching` page loads in under 2.5s LCP.

## 10. Edge Cases

1. **Empty history:** User has never watched anything. Rail is hidden. API returns `200` with `data: []`.
2. **All completed:** Every entry has `progress_pct >= 95`. Default rail filter excludes them. Dedicated page shows "Completed" tab.
3. **Episode soft-deleted:** Cursor references a deleted episode. System advances to the next available episode in the same anime. If none remain, the cursor row is removed.
4. **Anime removed from catalog:** Cursor references a non-existent anime. Entry is hidden from responses; no error thrown. A background job cleans up orphaned cursors.
5. **Network partition during heartbeat:** Client cannot reach the server. It stores the pending heartbeat in `localStorage` and retries on reconnect. Maximum drift: 5 minutes.
6. **Two devices heartbeating simultaneously:** Both read `version=14`. Device A writes first, `version=15`. Device B's write affects 0 rows, receives `409`. Device B re-reads `version=15`, merges its position, writes `version=16`.
7. **User scrubs to the beginning:** Heartbeat sends `position_seconds=0`. Server accepts; progress bar resets.
8. **User scrubs past the end:** Heartbeat sends `position_seconds > duration_seconds`. Server clamps to `duration_seconds` and returns `400` if the client doesn't clamp first.
9. **Rapid episode switch:** User finishes episode 3 and immediately starts episode 4. Two POSTs in quick succession. The second POST overwrites the first (idempotent on `anime_id`). Final state reflects episode 4 at position 0.

## 11. Error Handling

| Error Condition | User-Facing Message | Recovery Action | Log Level |
|-----------------|---------------------|-----------------|-----------|
| Unauthenticated request | "Please sign in to continue." | Redirect to login | warn |
| Heartbeat on missing cursor | "No cursor found. Starting from the beginning." | Client should POST to create cursor | info |
| Version conflict | "Syncing latest progress." | Re-read cursor, merge, retry | info |
| Rate limit exceeded | "Too many updates. Please wait." | Back off and retry after `Retry-After` | warn |
| Server error | "Unable to save progress. Your place may not be remembered." | Retry with exponential backoff | error |

## 13. Security Considerations

- All endpoints require a valid session; unauthenticated requests receive `401`.
- Users may only read and write their own `continue_watching` rows; user_id is derived from the session, never from the request body.
- Rate limit of 30 requests per 60 seconds per user prevents abuse (one heartbeat every 30s = 2/min, with headroom for episode switches).
- Heartbeat payload is validated: `position_seconds >= 0`, `duration_seconds > 0`, `progress_pct` computed server-side (never trusted from client).
- No PII in `continue_watching` beyond `user_id`; the table contains no user-generated content.
- Cache headers (`no-store`) prevent CDN or browser from serving stale cursor data.

## 14. Performance Requirements

- **Home-page rail query:** p95 < 200ms (indexed on `user_id, updated_at DESC`).
- **Heartbeat PATCH:** p95 < 100ms (single-row UPDATE with version check).
- **Dedicated page load:** LCP < 2.5s (SSR with streaming for authenticated user).
- **Rendering strategy:** SSR for the home page rail (personalized, no caching); CSR for the dedicated page's remove-action interactions.
- **Client JS budget:** Continue-watching rail adds < 5 KB gzipped to the home-page bundle.
- **Heartbeat overhead:** < 500 bytes per request; debounced to 30-second intervals.

## 15. Future Improvements

1. **Cross-device sync via WebSocket** — push cursor updates in real-time so a second device's rail refreshes without a page reload (M5).
2. **Smart resume** — skip intro/credits by detecting recurring position patterns across users (M6).
3. **Continue Watching for non-episode content** — extend to movies and OVAs with a unified cursor model (M4).
4. **Bulk progress reset** — allow users to "mark as unwatched" for a specific anime, removing the cursor and anonymizing watch history (M5).
5. **Progress sync with external trackers** — export continue-watching state to MyAnimeList, AniList, Kitsu via OAuth (M7).
