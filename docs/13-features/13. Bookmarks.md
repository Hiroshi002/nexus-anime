# Bookmarks — Nexus Anime

> **Audience:** Engineers (frontend + backend), Product, Design
> **Milestone:** M3
> **Owner:** Engineering
> **Status:** Draft

---

## 1. Purpose

Bookmarks are a user's personal watchlist. Each entry links a user to an anime and carries optional metadata: a personal note, a manual sort order, and a notification opt-in for new episodes. Bookmarks power the "My List" feature at `/bookmarks`, the bookmark toggle icon on every anime detail page, and the notification fan-out when a saved anime gets a new episode. Bookmarks are **private by default** — only the owner can see their own list.

## 2. Business Goals

- **Increase return visits** — users with items in their list have a reason to come back; target +12% 7-day retention for users with 3+ bookmarks.
- **Drive episode consumption** — new-episode notifications for bookmarked anime are a top re-engagement channel; target +20% episode-start rate from notifications.
- **Conversion funnel** — bookmarks correlate with subscription intent; track bookmark-to-premium conversion rate.
- **Catalog discovery** — a well-stocked watchlist reduces the "what to watch" paralysis that leads to session abandonment.

## 3. Functional Requirements

### 3.1 Happy Path

1. User navigates to an anime detail page and taps the bookmark icon.
2. System sends `POST /api/v1/users/me/bookmarks` with `{ animeId }`.
3. Icon fills (visual confirmation). The bookmark is added with `sortOrder = 0` and `notifyOnNewEpisode = true`.
4. User navigates to `/bookmarks` and sees a grid of bookmarked anime, sortable by "Date Added" or "Manual Order."
5. User can reorder items via drag-and-drop; each reorder sends `PATCH /api/v1/users/me/bookmarks/{animeId}` with a new `sortOrder`.
6. User can toggle `notifyOnNewEpisode` on a per-anime basis.
7. User can add a personal note (`note`, max 500 characters).
8. User can remove a bookmark (soft-delete); the icon returns to its outlined state.

### 3.2 Alternate Flows

1. **Duplicate prevention:** User taps the bookmark icon on an already-bookmarked anime. POST returns `409 Conflict` with `code: DUPLICATE_BOOKMARK`. The client treats this as idempotent (icon stays filled).
2. **Re-adding after removal:** User removes a bookmark, then adds it again. A new row is inserted (not restored from soft-delete) so `createdAt` reflects the actual "date added" time.
3. **Anime removed from catalog:** If the bookmarked anime is soft-deleted from the catalog, the bookmark remains but the UI shows "This anime is no longer available." If hard-deleted, a reconciliation job removes affected bookmarks.
4. **Very large bookmark list (500+ items):** Cursor pagination keeps response time flat. Sort by `sortOrder` is indexed. No full-list response.
5. **Toggling from detail page vs. list page:** Both use the same API. The detail page sends POST (add) or DELETE (remove); the bookmarks page additionally supports PATCH for note/sort changes.

### 3.3 Edge Cases

1. User with zero bookmarks sees an empty-state grid with a "Browse anime" CTA.
2. User tries to bookmark an anime that no longer exists (soft-deleted). POST returns `404 ANIME_NOT_FOUND`.
3. User rapidly taps the bookmark icon. POST is idempotent via `Idempotency-Key`; duplicate requests within 24 hours return the stored response.
4. Drag-and-drop race condition: user reorders two items simultaneously. Each PATCH is independent; the last PATCH wins (no cross-item constraint violation).
5. User imports a large list of anime (e.g., from MyAnimeList). Bulk-add endpoint may be needed if import exceeds 100 items; use batched POSTs.
6. User without a session taps bookmark. The client should redirect to login and queue the bookmark action for post-auth execution.

## 4. Non-Functional Requirements

- **Performance:** POST and DELETE p95 < 100ms; GET page p95 < 200ms; PATCH p95 < 100ms.
- **Availability:** 99.9% for reads and writes. Transient DB failures during POST should not corrupt the bookmark set.
- **Scalability:** Each user may have up to 500 active bookmarks in M3. Total bookmarks: ~500M at 1M users (average 500 per user, but heavy users may have more). Set a soft cap at 1000 per user and communicate it in the UI.
- **Accessibility:** Grid is a semantic list; reorder is keyboard-accessible (Alt+Up/Down); screen readers announce current position.
- **Localization:** Note field supports CJK; all UI strings externalized.
- **Security:** All endpoints require authentication; users may only access their own bookmarks; `userId` is derived from the session, never from the request body; rate limit 10 bookmark mutations per 60 seconds per user.

## 5. User Stories

- As a **logged-in user**, I want to save an anime to my list so that I can watch it later.
- As a **logged-in user**, I want to see all my saved anime in one place so that I can decide what to watch next.
- As a **logged-in user**, I want to reorder my list so that my priority shows appear first.
- As a **logged-in user**, I want to add a personal note to a bookmark so that I remember why I saved it.
- As a **logged-in user**, I want to opt out of new-episode notifications for a specific anime so that I am not spammed.
- As a **privacy-conscious user**, I want my bookmarks to be private so that other users cannot see what I plan to watch.

## 6. Acceptance Criteria

- [ ] POST `/api/v1/users/me/bookmarks` with a valid `animeId` returns `201` and the bookmark is visible in the list.
- [ ] POST with a duplicate `animeId` returns `409 DUPLICATE_BOOKMARK`.
- [ ] POST without `Idempotency-Key` returns `400 IDEMPOTENCY_KEY_REQUIRED`.
- [ ] POST with a replayed `Idempotency-Key` returns the original response byte-for-byte.
- [ ] DELETE `/api/v1/users/me/bookmarks/{animeId}` soft-deletes the bookmark; subsequent GET does not return it.
- [ ] PATCH `/api/v1/users/me/bookmarks/{animeId}` updates `note`, `sortOrder`, or `notifyOnNewEpisode`.
- [ ] GET `/api/v1/users/me/bookmarks?sort=sortOrder` returns bookmarks in the user's manual order.
- [ ] GET `/api/v1/users/me/bookmarks?sort=createdAt` returns bookmarks newest-first.
- [ ] GET `/api/v1/users/me/bookmarks/{animeId}` returns `200` if bookmarked, `404` if not.
- [ ] Unauthenticated requests receive `401`.
- [ ] User A cannot see or modify User B's bookmarks.
- [ ] `/bookmarks` page LCP < 2.5s.

## 10. Edge Cases

1. **Empty bookmark list:** User has no bookmarks. GET returns `200` with `data: { items: [] }`. Page shows an empty-state illustration with a "Browse anime" CTA.
2. **Anime soft-deleted from catalog:** Bookmark row remains. GET returns the bookmark with a flag `animeUnavailable: true`. The UI shows "This anime is no longer available" and disables the play button.
3. **Anime hard-deleted from catalog:** A reconciliation job identifies bookmarks with no matching anime and soft-deletes them. The next GET excludes these entries.
4. **Duplicate bookmark (idempotent POST):** POST with the same `animeId` returns `409 DUPLICATE_BOOKMARK`. The client should treat this as a success (icon stays filled).
5. **Re-add after removal:** User removes a bookmark, then adds it again. A new row is inserted (not restored) so `createdAt` reflects the actual re-add time. The partial unique constraint allows this because the old row has `deleted_at IS NOT NULL`.
6. **Very large bookmark list (500+ items):** Cursor pagination ensures each page resolves in constant time. The UI shows a count badge ("500+ items"). Sort by `sortOrder` is indexed.
7. **Rapid drag-and-drop reorder:** User drags item from position 1 to position 500. PATCH fires once on drop (not per intermediate position). The `sortOrder` value is set to the midpoint between neighbors (e.g., `sortOrder = (499 + 500) / 2 = 499.5`) to minimize re-sort cascades.
8. **Note with special characters:** Note contains emoji, CJK, or special characters. Stored as plain text; rendered through DOMPurify to prevent XSS. Max 500 characters enforced.
9. **Notification opt-out race:** User toggles `notifyOnNewEpisode` while a notification job is fanning out. The job reads the flag at the start of the fan-out; late toggles are respected for the next fan-out.
10. **Bulk import from external service:** User imports 200 anime from MyAnimeList. Client batches 10 POSTs per request. Rate limit (10/60s) is per-user; the import takes ~20 seconds. The UI shows progress.

## 11. Error Handling

| Error Condition | User-Facing Message | Recovery Action | Log Level |
|-----------------|---------------------|-----------------|-----------|
| Unauthenticated request | "Please sign in to save bookmarks." | Redirect to login | warn |
| Missing `Idempotency-Key` on POST | "Could not save. Please try again." | Client generates a new key and retries | info |
| `animeId` not found | "This anime is not available." | Refresh the page | info |
| Duplicate bookmark | "This anime is already in your list." | No-op (icon stays filled) | info |
| Note exceeds 500 characters | "Note is too long. Maximum 500 characters." | Client truncates or rejects | info |
| Rate limit exceeded | "Too many changes. Please wait." | Back off using `Retry-After` header | warn |
| Internal server error | "Unable to update your list. Please try again." | Retry with exponential backoff | error |

## 13. Security Considerations

- All endpoints require a valid session. Users may only access their own bookmarks; `userId` is derived from `session.sub`, never from the request body.
- POST requires an `Idempotency-Key` header. The key is stored in `@nexus/cache` with a 24-hour TTL. Replayed keys return the stored response byte-for-byte.
- Rate limit of 10 mutations per 60 seconds per user prevents abuse.
- `note` is plain text only; rendered through DOMPurify on the client to prevent XSS. Server-side validation enforces max 500 characters.
- Bookmark responses are `Cache-Control: private, max-age=30, must-revalidate` — never served from shared caches.
- The `bookmark_count` on the `anime` table is a denormalized counter. It is incremented/decremented in the same transaction as the bookmark insert/delete to prevent drift.
- No cross-user read surface in M3. Other users cannot see a user's bookmark list. The only public signal is the aggregate `bookmark_count` on the anime detail page.
- Logged events exclude the `note` content (PII-adjacent).

## 14. Performance Requirements

- **POST latency:** p95 < 100ms (single INSERT + `bookmark_count` UPDATE in one transaction).
- **DELETE latency:** p95 < 100ms (single UPDATE to `deleted_at` + `bookmark_count` UPDATE in one transaction).
- **GET page latency:** p95 < 200ms (cursor scan on `idx_bookmarks_user_id`).
- **PATCH latency:** p95 < 100ms (single-row UPDATE).
- **Page load:** `/bookmarks` LCP < 2.5s (SSR with streaming).
- **Rendering strategy:** SSR for the initial page load; CSR for drag-and-drop reorder and note editing.
- **Client JS budget:** Bookmarks page adds < 10 KB gzipped to the bookmarks bundle.
- **Cache invalidation:** On mutation, the handler invalidates `nexus:bookmarks:user:{userId}:list:*` and `nexus:bookmarks:user:{userId}:anime:{animeId}` in `@nexus/cache` synchronously before returning.

## 15. Future Improvements

1. **Public bookmark lists** — allow users to make their list public with a shareable URL (opt-in) (M5).
2. **Bookmark collections** — group bookmarks into named folders (e.g., "Weekend binge," "With friends") (M5).
3. **Bulk import** — one-click import from MyAnimeList, AniList, Kitsu via OAuth (M6).
4. **Bookmark recommendations** — suggest anime to add based on list composition (M6).
5. **Collaborative lists** — shared watchlists for families or friend groups (M7).
6. **Bookmark expiry** — auto-remove bookmarks inactive for 12+ months (with notice) (M8).
