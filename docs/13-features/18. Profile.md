# Profile — Nexus Anime

> **Audience:** Engineers, Product, Design
> **Milestone:** M4
> **Owner:** Product
**Status:** Draft

---

## 1. Purpose
The Profile feature is the user's public-facing identity on Nexus Anime: a profile page at `/users/{username}` displaying avatar, display name, bio, public statistics (watch count, bookmark count, join date), recent activity, and a watchlist. It exists to let users express themselves, to surface their taste socially, and to give returning users a personal home base that summarizes their journey. Own-profile views include editing controls; other users see a read-only public view governed by visibility settings.

## 2. Business Goals
- **Retention** — a personalized home base increases return frequency; target 8% lift in 30-day return rate for users who complete their profile (add avatar + bio).
- **Social/engagement** — public profiles and watchlists drive organic discovery and follower-style engagement; target 5% increase in profile→bookmark cross-navigation.
- **Trust** — visible, honest stats and watched history signal an active, real community, reinforcing premium positioning.
- **Data ownership** — user-editable identity (avatar, bio, visibility) hands control to the user and reduces support tickets around "make this private."

## 3. Functional Requirements

### 3.1 Happy Path
1. A logged-in user navigates to their own `/profile` summary hub (a canonical alias for `/users/{self.username}`).
2. The user edits their display name, bio, avatar, and watchlist visibility via a Settings-integrated form; changes persist and immediately reflect publicly.
3. Any visitor (signed in or not) views `/users/{username}` and sees the avatar, display name, bio, public stats, recent activity, and — if visibility allows — the user's watchlist.
4. The profile shows recent activity (comments, ratings, bookmarks) in a chronological feed, cursor-paged.
5. The user toggles watchlist/profile visibility between `public`, `private`, and (future) `friends`.

### 3.2 Alternate Flows
1. **Username change** — user updates their canonical slug; old username returns a 301 redirect to the new one for 90 days, after which it becomes available. URLs with the old slug in comments/ratings still resolve to the redirected profile.
2. **Private profile** — a viewer hitting a private profile sees a "This profile is private" empty state without leaking stats or activity.
3. **New user with no activity** — profile page renders with zeroed stats and an empty-state prompt ("Start watching and bookmarking to fill your profile").
4. **Avatar upload** — user uploads a new avatar; client validates size (< 2 MB) and format (JPEG/PNG/WebP); server re-encodes, generates two sizes, and stores via Cloudflare R2.
5. **Bio with links** — bio text is plain text only; URLs are auto-linked client-side, and new-user bios with links are held for moderation per anti-spam rules.

### 3.3 Edge Cases
1. Viewing a nonexistent username → `USER_NOT_FOUND` (404) with a "No user found" state.
2. Viewing one's own profile via `/users/{self.username}` → same as `/profile` with edit controls.
3. Profile visibility changed from public to private → cached public pages invalidated; existing deep links now show the private state.
4. Avatar upload exceeding size/format limits → client-side rejection first; server-side Zod validation as backstop.
5. Username change while another session holds the old URL → 301 redirect resolves transparently.

## 4. Non-Functional Requirements
- **Performance:** Profile page p95 < 300ms (SSR with ISR revalidate 60s); activity feed p95 < 200ms; avatar image served via `next/image` + R2 loader with < 100ms TTFB.
- **Availability:** 99.9% read path; edit path tolerates brief degradation.
- **Scalability:** 2k profile reads/sec sustained; 50 avatar uploads/min burst.
- **Accessibility:** Avatar has meaningful `alt`; stats use semantic markup; edit form is keyboard-navigable with visible focus; color contrast ≥ 4.5:1. WCAG 2.1 AA.
- **Localization:** Display names and bio support CJK/emoji; timestamps localized; future RTL via logical properties.
- **Security:** Public/private scoping enforced server-side; bio sanitized (DOMPurify) before render; avatar uploads validated for type/size; username uniqueness enforced at DB; old-username redirects prevent impersonation.

## 5. User Stories
- As a **logged-in user**, I want to set my avatar, display name, and bio so that I'm recognizable to others.
- As a **logged-in user**, I want to control who sees my watchlist and stats so that I can keep my taste private.
- As a **visitor**, I want to view someone's profile so that I can discover what they've watched and bookmarked.
- As a **returning user**, I want a summary of my recent activity so that I can pick up where I left off.
- As a **moderator**, I want bios sanitized and new-user link bios flagged so that spam is contained.

## 6. Acceptance Criteria
- [ ] `/users/{username}` renders avatar, display name, bio, watch count, bookmark count, and join date for a public profile.
- [ ] A private profile returns a "This profile is private" state to non-owners without leaking any stats or activity.
- [ ] Editing display name/bio/avatar via the own-profile form persists and reflects immediately on the public page.
- [ ] Changing username issues a 301 from the old slug to the new one for 90 days; old URLs in comments/ratings still resolve.
- [ ] Avatar upload rejects >2 MB and non-image formats client-side; server Zod validation rejects as backstop.
- [ ] Bio is rendered sanitized (no script injection); auto-linked URLs open in a new tab with `rel="noopener"`.
- [ ] A profile with zero activity renders the empty-state prompt, not a broken/blank feed.
- [ ] Nonexistent username returns `USER_NOT_FOUND` (404) with a friendly state.

## 10. Edge Cases
1. **User with no activity** — stats are 0; activity feed renders empty-state CTA; page still loads (no 404).
2. **Private profile viewing** — server detects viewer ≠ owner AND visibility ≠ public → returns minimal envelope with `private: true`; UI renders `PrivateProfileGate`.
3. **Username change / URL breakage** — old slug inserted into `username_redirects`; GET on old slug issues 301 to new canonical URL; 90-day TTL then row is deleted and slug becomes available.
4. **Avatar upload (size/format)** — client validates < 2 MB and JPEG/PNG/WebP; server re-validates via Zod + magic-byte check; rejects with `FIELD_INVALID`. Server re-encodes to 256px/64px variants stored on R2.
5. **Bio with links (spam)** — bio stored as plain text; rendered through DOMPurify; new accounts (< 7 days) with URLs in bio are flagged to moderation queue.
6. **Concurrent username claim** — UNIQUE constraint on `username` (citent); second writer gets `USERNAME_TAKEN` (409).
7. **Deleted user** — profile returns `USER_NOT_FOUND` (404); activity feed for that user is hidden; comments remain attributed to a "deleted user" placeholder.
8. **Self-view via public URL** — `/users/{self.username}` detects ownership and renders edit controls identical to `/profile`.

## 13. Security Considerations
- **OWASP A01 (Broken Access Control)** — public/private scoping enforced server-side; private fields (email, private stats) never serialized to non-owners.
- **OWASP A03 (Injection / XSS)** — bio sanitized with DOMPurify; rendered as plain text with safe auto-linking; no raw HTML.
- **OWASP A05 (Misconfiguration)** — avatar upload validates MIME via magic bytes, not extension; size capped; stored on R2 with private ACL + signed URLs.
- **Impersonation** — old-username redirects are one-way and time-limited; usernames are unique and case-insensitive (citent); reserved usernames block squatting.
- **Spam** — new-user bios with links flagged to moderation queue; rate-limited edits (20/min).
- **PII** — email and private stats never appear in public profile responses; analytics use hashed IDs.

## 14. Performance Requirements
- Profile page p95 < 300ms (SSR, ISR revalidate 60s; cache key `nexus:user:{id}:profile`).
- Activity feed (first page) p95 < 200ms; cursor-paged.
- Avatar served via `next/image` + R2 loader; TTFB < 100ms; variants 256/64 px cached at edge.
- Edit mutations p95 < 250ms; write-through invalidates profile + activity caches.
- Rendering strategy: profile shell is SSR; activity feed is a Suspense boundary with skeleton; edit form is a client island.
- Client bundle budget for profile edit island: < 15 kB gzipped.
