# Watch History — Nexus Anime

> **Audience:** Engineers (frontend + backend), Product, Design, Legal (GDPR)
> **Milestone:** M3
> **Owner:** Engineering
> **Status:** Draft

---

## 1. Purpose

Watch History is an append-only log of every episode a user watches. It powers the `/history` page (a paginated, filterable list of past watches) and serves as the source of truth for `continue_watching`, popularity scoring, and future recommendations. It is the largest table in the system and is designed for high-write throughput and time-scoped reads.

## 2. Business Goals

- **Engagement** — a visible history encourages re-watching and completionist behavior; target +8% re-watch rate.
- **Personalization** — watch history is the primary signal for the M3 recommendation engine; better recs drive +12% click-through.
- **GDPR compliance** — anonymization-on-erasure satisfies right-to-erasure requests while preserving aggregate analytics.
- **Content strategy** — aggregate history drives view-count rankings, trending detection, and licensing ROI analysis.

## 3. Functional Requirements

### 3.1 Happy Path

1. User finishes (or pauses) an episode; the client fires a POST to record a watch event.
2. System validates the payload, derives `user_id` from the session, and appends a row to `watch_history`.
3. User navigates to `/history` to see their full watch history, cursor-paginated and sorted by `watched_at DESC`.
4. Each entry shows: anime title, episode number, episode title, watched date, completion percentage, device.
5. User can filter by date range (`from`, `to`) and by anime.
6. User can delete individual entries or clear all history (GDPR flow with `confirm=true`).

### 3.2 Alternate Flows

1. **Idempotent replay:** Client retries a POST with the same `event_id`. System returns `200` with the original row; no duplicate is created.
2. **Offline buffering:** Client queues watch events while offline; on reconnect, events are posted with the original `watched_at` timestamp.
3. **Date range filter with no results:** System returns `200` with `data: []` and `pagination.has_more = false`.
4. **Clearing history (GDPR):** User clicks "Clear All History." System requires explicit confirmation, then anonymizes all rows (sets `user_id = NULL`, nulls PII-adjacent fields).
5. **Very long history (10k+ entries):** Cursor pagination keeps response size bounded; response time stays flat regardless of total history size.

### 3.3 Edge Cases

1. User with zero watch history sees an empty state with a "Start watching" prompt.
2. History spanning several years — date-range filter should handle wide ranges efficiently (partition pruning on `watched_at`).
3. Concurrent POSTs from multiple devices with the same `event_id` — the second insert hits a unique constraint and returns `200` with the original row.
4. `watched_at` is in the future (client clock skew). System clamps to `now()` or rejects with `400`.
5. Episode referenced in a watch event is later soft-deleted. The history row is retained; the UI shows the episode as "Unavailable" if the episode is unplayable.
6. User watches the same episode multiple times. Each watch is a separate append (no dedup); the history shows all occurrences.
7. Export request (GDPR portability). User can download their history as JSON/CSV from `/history/export`.
8. User deletes account. Watch history is anonymized (not hard-deleted); aggregate analytics retained.

## 4. Non-Functional Requirements

- **Performance:** POST p95 < 50ms (append-only insert); GET page p95 < 150ms (cursor-scan on indexed `(user_id, watched_at DESC)`).
- **Availability:** 99.9% for reads; 99.5% for writes (stale reads acceptable; lost heartbeats degrade UX but are non-critical).
- **Scalability:** At 1M active users, the table reaches 500M+ rows/year. Partitioned by `watched_at` (monthly). Inserts: ~16k/sec peak.
- **Accessibility:** History list is a semantic `<ul>` with `aria-label` per entry; date filter is a labeled form; screen readers announce entry count.
- **Localization:** Dates rendered with `Intl.DateTimeFormat`; string externalization for all UI labels.
- **Security:** All endpoints require authentication; users may only read/write their own history; rate limit 30 writes per 60 seconds per user.

## 5. User Stories

- As a **logged-in user**, I want to see a list of episodes I have watched so that I can track my progress.
- As a **logged-in user**, I want to filter my history by date or anime so that I can find a specific watch event.
- As a **logged-in user**, I want to remove a single entry so that I can correct accidental watches.
- As a **logged-in user**, I want to clear my entire history so that I can exercise my right to erasure.
- As a **privacy-conscious user**, I want to know that clearing history anonymizes rather than deletes my data so that aggregate stats are preserved transparently.

## 6. Acceptance Criteria

- [ ] POST `/api/v1/users/me/watch-history` appends a watch event and returns `201`.
- [ ] POST with a duplicate `event_id` returns `200` with the original row and does not create a duplicate.
- [ ] GET `/api/v1/users/me/watch-history?limit=20` returns the 20 most recent entries with cursor pagination.
- [ ] GET with `from` and `to` query parameters returns only entries in the date range.
- [ ] GET with `anime_id` filter returns only entries for that anime.
- [ ] DELETE `/api/v1/users/me/watch-history/{id}` hard-deletes the entry.
- [ ] DELETE `/api/v1/users/me/watch-history?confirm=true` anonymizes all rows for the user.
- [ ] Unauthenticated requests receive `401`.
- [ ] User A cannot see or delete User B's history entries.
- [ ] `/history` page LCP < 2.5s.

## 10. Edge Cases

1. **Empty history:** User has never watched anything. GET returns `200` with `data: []`. Page shows an empty-state illustration.
2. **Very long history (10k+ entries):** Cursor pagination ensures each page resolves in constant time. Total count is not computed on subsequent pages to avoid a full-table scan.
3. **History export (GDPR portability):** GET `/history/export?format=json|csv` returns a downloadable file. Rate-limited to 5 exports per 60 seconds per user. Export contains only the user's own rows; anonymized rows (pre-erasure events) are excluded.
4. **Partial watch vs. completed:** `completion_pct = 0` means the user opened the episode but didn't watch anything; `completion_pct >= 95` is treated as "completed" in aggregate metrics. Both are displayed in history.
5. **Offline events:** Client queues events locally with `watched_at` set to the local timestamp. On reconnect, events are POSTed. The server validates `watched_at` is not in the future (tolerance: 1 minute for clock skew).
6. **Future-dated watch event:** `watched_at > now() + 60s`. Server returns `400` with code `VALIDATION_ERROR` and explanation.
7. **Race condition on bulk delete:** Two simultaneous anonymization requests. The second request finds no rows to anonymize (already anonymized) and returns `anonymized_count: 0`.
8. **Partition boundary:** Query spans a partition boundary (`from` in January, `to` in February). The query planner scans both partitions; performance is acceptable due to partition pruning.
9. **Deleted episode reference:** `episode_id` references a soft-deleted episode. History row is retained; the UI shows the episode as "Unavailable" with no play link.
10. **Deleted anime reference:** `anime_id` references a hard-deleted anime (rare; only if catalog cleanup was aggressive). History row is retained; the UI shows "[Removed Anime]".

## 11. Error Handling

| Error Condition | User-Facing Message | Recovery Action | Log Level |
|-----------------|---------------------|-----------------|-----------|
| Unauthenticated request | "Please sign in to view your history." | Redirect to login | warn |
| Invalid `event_id` format | "Could not save watch event. Please try again." | Client generates a new UUID | info |
| `watched_at` in the future | "Watch time cannot be in the future." | Client should send `watched_at = null` | warn |
| Rate limit exceeded | "Saving is temporarily throttled. Please wait." | Back off using `Retry-After` header | warn |
| Row not found (DELETE) | "This entry no longer exists." | Refresh the list | info |
| Bulk anonymization without `confirm` | "This action requires confirmation." | Re-open confirmation dialog | info |
| Internal server error | "Unable to load history. Please try again." | Retry with exponential backoff | error |

## 13. Security Considerations

- All endpoints require a valid session. Users may only access their own `watch_history` rows (derived from `session.sub`).
- `user_id` is NEVER accepted from the request body on POST; it is always the authenticated session user.
- Rate limit of 30 writes per 60 seconds per user prevents write flooding.
- Bulk anonymization requires `confirm=true`; defense against accidental invocation.
- The export endpoint is rate-limited to 5 requests per 60 seconds per user to prevent enumeration or excessive data export.
- GDPR: anonymization sets `user_id = NULL`, nulls `device`/`os`/`browser`/`country`/`app_version`. The `is_anonymized` flag is set. The row is retained for aggregate analytics.
- No raw SQL in application code; all access is through the repository layer.
- Watch history responses are `Cache-Control: private, no-store` — never served from shared caches.
- Logged events exclude PII-adjacent fields (OS, browser, country, app_version) from application logs.

## 14. Performance Requirements

- **POST latency:** p95 < 50ms (append-only INSERT into a partitioned table).
- **GET page latency:** p95 < 150ms (bounded by cursor scan on `idx_watch_history_user_watched_at`).
- **Bulk anonymization:** completes in < 5 seconds for a user with 10k entries (single bulk UPDATE).
- **Page load:** `/history` LCP < 2.5s (SSR with streaming).
- **Rendering strategy:** SSR for the initial history page load; CSR for filter interactions and infinite scroll.
- **Client JS budget:** Watch-history page adds < 8 KB gzipped to the history bundle.
- **Partitioning:** Monthly range partitions on `watched_at`. Old partitions are archived to cold storage after 90 days.
- **Aggregation:** Raw rows older than 90 days are rolled up into daily aggregates (`anime_id`, `country`, `device`, `completion_bucket`, `count`, `avg_completion_pct`); raw rows are purged by a batch job.
