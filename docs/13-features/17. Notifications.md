# Notifications — Nexus Anime

> **Audience:** Engineers, Product, Design
> **Milestone:** M4 (in-app bell + read/unread); M5 (Server-Sent Events); M7 (push + email digest)
> **Owner:** Engineering
> **Status:** Draft

---

## 1. Purpose
The Notifications feature gives authenticated users a single, in-app inbox for event-driven signals — new episodes, comment replies, recommendation updates, and system announcements. It exists to bring users back to relevant moments without relying on external channels, to close the loop on social interactions (replies), and to give the platform a controlled channel for trustworthy system messaging. Real-time delivery ships via polling in M4 and upgrades to Server-Sent Events in M5.

## 2. Business Goals
- **Retention** — bring users back at meaningful moments; target 10% lift in 7-day return rate for users who engage with a notification within 1 hour.
- **Engagement** — close the social loop on comments (reply → reply); target 8% increase in comment reply chains within 60 days.
- **Trust** — a trusted, in-app system channel reduces reliance on email and limits phishing surface compared with outbound mail.
- **Operational safety** — system announcements (scheduled maintenance, content removals) reach active users in-product, reducing support tickets by an estimated 5%.

## 3. Functional Requirements

### 3.1 Happy Path
1. A logged-in user sees a notification bell in the top nav showing an unread badge count.
2. The user opens the bell dropdown; the latest notifications (newest first, cursor-paged) render with title, body excerpt, timestamp, and a read/unread indicator.
3. The user taps a notification; the UI marks it read, decrements the badge, and navigates to the target (episode, comment thread, anime page).
4. The user dismisses a single notification or marks all as read via a single action.
5. Notification preferences (per-type opt-out) are configurable in Settings.

### 3.2 Alternate Flows
1. **Polling (M4)** — client polls `/api/v1/notifications?unread=true` every 60s (and on window focus) to refresh the badge; SSE is unavailable.
2. **SSE (M5)** — once available, client opens an EventSource to `/api/v1/notifications/stream`; server pushes `notification.created` and `unread_count` events; polling is a graceful fallback.
3. **Deep link** — user follows a notification link while signed out; after sign-in, they land on the notification's target with the item marked read.
4. **Deleted content target** — notification points to an episode/anime that was soft-deleted; the target view shows a "This content is no longer available" state and the notification body is preserved for record.
5. **Bulk burst** — a new season drops and thousands of users are notified; creation is rate-limited per-recipient and queued; users see notifications within a bounded delay (≤30s).

### 3.3 Edge Cases
1. Notification for deleted content → rendered with a "unavailable" badge; tapping opens a graceful empty state, does not 404.
2. Overflow unread count — badge caps display at "99+" while the real integer is preserved internally.
3. Rapid read toggles → debounced PATCH; last-write-wins on `read_at`.
4. Notification for a private/removed comment target → anonymous-safe body, target resolves to a generic "comment unavailable" view.
5. User opts out of a notification type mid-flight → creation respects the preference snapshot at creation time; updates apply to future events.

## 4. Non-Functional Requirements
- **Performance:** Badge count fetch p95 < 80ms; notification list p95 < 150ms on first page; SSE event-to-render < 500ms in M5.
- **Availability:** 99.9% read path; notification creation is best-effort (non-blocking for the triggering action); lost notifications are acceptable within a bounded window.
- **Scalability:** Support 10k concurrent SSE connections (M5); 500 notification creations/sec at season-drop scale; list reads 1k req/s.
- **Accessibility:** Bell is a button with `aria-label` reflecting uncount ("Notifications, 3 unread"); list is an `listbox`/`feed`; new announcements use `aria-live="polite"`. WCAG 2.1 AA.
- **Localization:** All notification templates externalized; timestamps rendered in user locale; future RTL support via CSS logical properties.
- **Security:** Per-user scoping enforced server-side (no IDOR); SSE stream authenticated via cookie; no PII in cross-user announcements; system notifications signed/validated as originating from the platform.

## 5. User Stories
- As a **logged-in user**, I want to know when a new episode drops so that I can watch it promptly.
- As a **logged-in user**, I want to be told when someone replies to my comment so that I can continue the conversation.
- As a **logged-in user**, I want to clear or dismiss notifications so that my inbox stays tidy.
- As a **logged-in user**, I want to opt out of notification types I don't care about so that I'm not interrupted.
- As an **admin**, I want to send a system announcement to all users (or a segment) so that I can communicate scheduled maintenance or policy changes.

## 6. Acceptance Criteria
- [ ] Unread badge reflects the true unread count for the session user within 2 seconds of a new notification (M5 SSE) or on next poll (M4).
- [ ] Tapping a notification marks it read, decrements the badge, and navigates to the correct target.
- [ ] A user can mark all notifications as read; badge returns to 0; all rows have `read_at` set.
- [ ] A user can delete a single notification; it disappears and the badge decrements if it was unread.
- [ ] Notification for deleted content renders an "unavailable" badge and opens a graceful empty state without a 404.
- [ ] Opting out of a type in Settings prevents future notifications of that type; existing notifications are unaffected.
- [ ] Unauthenticated access to any notification endpoint returns `UNAUTHORIZED` (401); cross-user reads return empty/404 (no IDOR).
- [ ] Unread count badge displays "99+" for counts ≥100 while the real value is preserved.
- [ ] At season-drop scale, notifications are delivered to all targeted users within 30 seconds.

## 10. Edge Cases
1. **Notification for deleted content** — FK is unenforced (logic-level) so notification persists; resolver marks `targetAvailable = false` and UI shows "unavailable" badge with a preserved body.
2. **Unread-count overflow** — stored as integer; badge caps display at "99+", never wraps or overflows the nav.
3. **Rapid read toggles** — client debounces (300ms); server applies last-write-wins on `read_at`; badge is recomputed from `COUNT(read_at IS NULL)`.
4. **Notification to a deleted/suspended user** — creation is skipped silently (no orphan rows); logged at info level.
5. **Burst on season drop** — creation is funneled via a queue (Redis list or job table); per-recipient deduplication ensures one notification per user per event.
6. **SSE reconnect (M5)** — client auto-reconnects with `Last-Event-ID`; server replays missed events since that ID.
7. **Preference opt-out mid-flight** — preference is snapshotted at event-handling time, not delivery time; changing preferences affects only future events.
8. **Concurrent read-all** — atomic UPDATE; badge recount reconciles to 0; no partial states.

## 11. Error Handling

| Error Condition | User-Facing Message | Recovery Action | Log Level |
|-----------------|---------------------|-----------------|-----------|
| SSE connection lost | (silent) | Auto-reconnect with backoff; fall back to 60s polling | warn |
| Load list fails | "Couldn't load notifications. Pull to retry." | Retry; cache last successful page | error |
| Mark read fails | "Couldn't update. Try again." | Optimistic UI rolls back | warn |
| Notification not found / cross-user | "Notification not found." | 404; no IDOR signal | warn |
| Announcement send (admin) non-unicast | — (success path) | Validation rejects invalid segments | info |

## 13. Security Considerations
- **OWASP A01 (Broken Access Control)** — every query scoped by `session.user.id`; `{id}` paths re-checked for ownership; cross-user reads return generic 404 (no enumeration).
- **OWASP A03 (Injection)** — Zod-validated inputs; notification bodies treated as plain text in the UI (no HTML rendering).
- **IDOR** — unread count and list never expose another user's data; SSE stream tied to the authenticated cookie only.
- **System announcements** — admin-only creation; all sends logged immutably to `audit_log`.
- **Spoofing** — users cannot create notifications manually; only server-side event handlers and admin endpoints mutate the table.
- **PII** — titles/bodies use templated phrasing; no user-emitted free text leaks across users.

## 14. Performance Requirements
- Badge count fetch p95 < 80ms (cached per-user, TTL 30s).
- Notification list (first page, 20 items) p95 < 150ms.
- SSE event-to-render (M5) p95 < 500ms end-to-end under 5k concurrent connections.
- Creation event ingestion sustains 500 notifications/sec at season-drop scale with queue-backed durability.
- Rendering: bell is a client island (lazy, < 6 kB); list uses Suspense skeleton; infinite scroll via cursor pagination.
- Polling (M4) degrades gracefully: only the lightweight unread-count endpoint is polled every 60s; full list fetched on dropdown open.
