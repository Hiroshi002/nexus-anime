# Admin — Nexus Anime

> **Audience:** Engineers, Product, Trust & Safety, Leadership
> **Milestone:** M4 (dashboard shell, user management, moderation queue); M5 (analytics overview); M7 (full content CRUD, advanced analytics)
> **Owner:** Engineering
> **Status:** Draft

---

## 1. Purpose
The Admin feature is the internal operations surface for Nexus Anime: a role-gated dashboard at `/admin` for user management, content management, moderation, and analytics. It exists to let the team operate the platform safely — to handle abuse, manage the catalog, respond to reported content, and understand platform health — without touching production systems directly. Access is restricted to the `admin` role (and `moderator` for the moderation queue), enforced by RBAC, and every action is logged immutably.

## 2. Business Goals
- **Safety** — fast, auditable moderation reduces the lifespan of abusive content; target median report-to-action time < 4 hours.
- **Operational efficiency** — self-serve user/content management reduces engineering toil for routine operations by an estimated 70%.
- **Trust** — immutable audit logs and 2FA-gated access create accountability and support compliance reviews.
- **Data-driven decisions** — an analytics overview (DAU, top content, conversion funnel) gives leadership a single source of truth for platform health.

## 3. Functional Requirements

### 3.1 Happy Path
1. An admin navigates to `/admin`; the route guard verifies the `admin` role and 2FA enrollment; the dashboard shell loads with summary KPIs.
2. In User Management, the admin searches users by email/username, views details, bans/unbans, or deletes an account (with confirmation).
3. In Content Management, the admin creates/edits/soft-deletes anime and episodes, or imports from upstream IDs (TMDB/AniList).
4. In the Moderation Queue, the admin (or moderator) reviews reported comments and flagged content, then approves, removes, or escalates.
5. In Analytics, the admin views DAU/MAU, top content, conversion funnel, and revenue metrics (M5+).

### 3.2 Alternate Flows
1. **Moderator role** — a `moderator` can access only the Moderation Queue; other sections return `FORBIDDEN` (403).
2. **Self-action guard** — an admin cannot ban/delete their own account from the UI; the action is rejected server-side.
3. **Concurrent edits** — two admins editing the same anime row: optimistic concurrency via `version`; second writer gets `CONFLICT` (409) with a refresh prompt.
4. **Large user list** — search/pagination is cursor-based; exports are async for lists >10k rows.
5. **Admin action on own content** — allowed but flagged in the audit log with `self_action: true` for compliance review.

### 3.3 Edge Cases
1. Admin accidentally attempts to ban themselves → blocked server-side with a clear message.
2. Concurrent admin edits on the same row → `CONFLICT` (409); refresh and re-apply.
3. Very large user list (100k+) — search uses indexed columns; pagination is cursor-based; exports are async.
4. Audit log retention — logs are append-only; retention policy (e.g., 1 year hot, 7 years cold) is configurable.
5. Admin action on their own content — allowed but flagged with `self_action: true`.

## 4. Non-Functional Requirements
- **Performance:** Dashboard KPIs p95 < 500ms (materialized/aggregated); user search p95 < 300ms; moderation queue p95 < 200ms.
- **Availability:** 99.9% read path; write path tolerates brief degradation; audit logging is best-effort (failure does not block the action but alerts).
- **Scalability:** Support 100k+ user search via indexed queries; analytics aggregations pre-computed on a schedule.
- **Accessibility:** Dashboard is keyboard-navigable; tables have proper headers; destructive actions require confirmation. WCAG 2.1 AA.
- **Localization:** Admin UI is English-only at launch (internal tool); future i18n if needed.
- **Security:** Admin route guard (RBAC + 2FA); optional IP allowlist; all actions logged immutably; no mass operations without confirmation; rate-limited writes; CSP strict.

## 5. User Stories
- As an **admin**, I want a single dashboard so that I can see platform health at a glance.
- As an **admin**, I want to search, ban, and delete users so that I can respond to abuse quickly.
- As an **admin**, I want to manage the anime catalog so that the content stays accurate and fresh.
- As a **moderator**, I want a moderation queue so that I can review reported content efficiently.
- As an **admin**, I want every action logged immutably so that we have accountability and can audit changes.
- As a **leadership stakeholder**, I want analytics overviews so that I can make data-driven decisions.

## 6. Acceptance Criteria
- [ ] `/admin` is inaccessible to non-admins (returns `FORBIDDEN` (403)); inaccessible to admins without 2FA enrollment (redirects to 2FA setup).
- [ ] User search returns results p95 < 300ms for 100k+ users; pagination is cursor-based.
- [ ] Banning a user invalidates their sessions immediately and prevents sign-in; unbanning restores access.
- [ ] Deleting a user requires typed confirmation; follows the same grace-period + purge flow as self-service deletion.
- [ ] An admin cannot ban or delete their own account; the action is rejected server-side with a clear message.
- [ ] Concurrent edits to the same row return `CONFLICT` (409) with a refresh prompt.
- [ ] Every admin action is written to `audit_log` with actor, action, target, timestamp, and IP hash.
- [ ] Moderators can access only the Moderation Queue; other sections return `FORBIDDEN`.
- [ ] Reported content can be approved, removed, or escalated; the reporting user is notified of the outcome.
- [ ] Audit logs are append-only and cannot be modified or deleted via the API.

## 10. Edge Cases
1. **Admin bans themselves** — server rejects with `CANNOT_SELF_BAN` (403); UI disables the button for the current user.
2. **Concurrent admin edits** — `version` column on `anime`; UPDATE with `WHERE version = :expected`; mismatch returns `CONFLICT` (409); client refreshes.
3. **Very large user list (100k+)** — search is cursor-based on indexed columns; exports are async (queue-backed); UI shows a capped preview with a "download full report" action.
4. **Audit log retention** — append-only; hot retention 1 year (queryable), cold archival to object storage after; configurable via env.
5. **Admin action on own content** — allowed (e.g., editing an anime they created) but flagged `self_action: true` in `audit_log` for compliance review.
6. **Moderator accessing admin-only routes** — `requireRole('admin')` returns `FORBIDDEN` (403); UI hides those sections.
7. **Banned user with active session** — ban deletes all `user_sessions` rows; subsequent requests return `UNAUTHORIZED`; middleware blocks access.
8. **Report on already-deleted content** — report remains for record; resolver sees a "content no longer available" state; can still mark resolved with a note.
9. **IP allowlist mismatch** — if enabled, requests from non-allowed IPs return `FORBIDDEN` before RBAC; logged at warn level.

## 13. Security Considerations
- **OWASP A01 (Broken Access Control)** — admin routes guarded by `requireRole('admin')`; moderators scoped to moderation only; self-ban/delete blocked server-side.
- **OWASP A02 (Cryptographic failures)** — 2FA (TOTP) required for admin accounts; sessions use the same hardened cookie policy; IP allowlist optional.
- **OWASP A03 (Injection)** — all inputs Zod-validated; Drizzle parameterized queries; no raw SQL in admin handlers.
- **OWASP A05 (Misconfiguration)** — admin routes excluded from public sitemap/robots; strict CSP; no caching of admin responses at the edge.
- **Auditability** — every admin action written immutably to `audit_log` (append-only; no UPDATE/DELETE via app); logs include actor, action, target, timestamp, IP hash, and payload hash.
- **Mass operations** — bulk delete/ban requires typed confirmation and is rate-limited; no single-click mass actions.
- **PII** — admin views may show user email/PII; access is role-gated and logged; exports are signed and expiring.
- **Accountability** — shared admin accounts are prohibited; each admin uses their own credentials; all actions tied to an individual actor.

## 14. Performance Requirements
- Dashboard KPI tiles p95 < 500ms (pre-computed aggregates refreshed every 5 minutes; cache `nexus:admin:overview` TTL 60s).
- User search p95 < 300ms on 100k+ users (indexed `email`/`username`/`role`; cursor pagination).
- Moderation queue p95 < 200ms (indexed on `(status, created_at)`).
- Analytics p95 < 1s for 30-day windows (materialized rollups; cache TTL 300s).
- Rendering strategy: admin shell is SSR (no caching); KPI tiles use Suspense with skeleton; tables use cursor-paged client fetching.
- Client bundle budget for admin app: < 80 kB gzipped (admin-only; not loaded by regular users).
