# Recommendations — Nexus Anime

> **Audience:** Engineers (frontend + backend + ML), Product, Design
> **Milestone:** M3 (content-based) / M6+ (collaborative filtering)
> **Owner:** Engineering (ML)
> **Status:** Draft

---

## 1. Purpose

Recommendations serve personalized anime suggestions to users. The feature includes four surfaces: a personalized landing rail ("Recommended for You"), a global trending shelf, a "More Like This" shelf on anime detail pages, and a "Continue Watching" shelf. The M3 algorithm uses content-based filtering (genre overlap, studio overlap, popularity). A cold-start fallback ensures new users still see relevant content.

## 2. Business Goals

- **Increase catalog discovery** — users who see personalized recs browse 2x more titles; target +15% unique-anime-per-session.
- **Drive episode starts** — recommendations drive directly to the player; target +10% conversion from rec-tap to play.
- **Reduce churn** — users who engage with recs have higher LTV; target +6% 90-day retention.
- **Balance personalization with serendipity** — 20% of recs should be "exploration" (titles outside the user's top genres) to avoid filter bubbles.

## 3. Functional Requirements

### 3.1 Happy Path

1. Authenticated user lands on the home page.
2. System sends `GET /api/v1/recommendations?limit=10` with the user's session token.
3. Server computes (or serves cached) recommendations using genre-overlap + studio-overlap + popularity weighted blend.
4. Each item includes a `reason` (e.g., "Because you watched Dandadan") rendered as a contextual badge.
5. User taps an item to navigate to the anime detail page.
6. On the detail page, the "More Like This" shelf is populated by `GET /api/v1/recommendations/similar/{animeId}`.
7. Trending shelf on home is populated by `GET /api/v1/recommendations/trending?season=summer&year=2026`.

### 3.2 Alternate Flows

1. **New user (cold start):** User has no watch history. Personalized endpoint returns trending anime with `reason: "fallback"`. UI renders the shelf as "Trending Instead."
2. **Homogeneous taste:** User watches only one genre. Recs are dominated by that genre. The system injects 20% "exploration" items (highly-rated anime from adjacent genres).
3. **Very active user:** User has watched 500+ anime. Compute cost is high; the system relies on precomputed recommendations (refreshed weekly) rather than real-time computation.
4. **Trending with optional auth:** User is not authenticated. Trending endpoint returns global trending. User is authenticated. Trending endpoint applies genre-boost based on the user's top genres.
5. **Similar anime for niche title:** The source anime has few genre/studio overlaps. The endpoint falls back to popular anime of the same type (TV/movie/OVA) with `reason: "trending"`.

### 3.3 Edge Cases

1. **Empty catalog:** No anime exist. Endpoints return `200` with `data: []`. Home page hides the shelf.
2. **User with watchlist but no completed watches:** The system uses bookmarks as a weak signal (weight 0.1) for personalization.
3. **Anime deleted after recommendation list is computed:** The response includes an ID that no longer exists. The client should gracefully hide or refetch on tap.
4. **Recommendation freshness:** Precomputed recommendations are recomputed weekly. A user who watches an anime on Monday may see it recommended on Tuesday (since the list doesn't update daily). Acceptable staleness: 7 days.
5. **Recommendations for anime the user has already completed:** The system excludes anime present in `continue_watching` with `is_completed = true` or in the user's watchlist.
6. **Rapid genre evolution:** User binge-watches a new genre. The weekly recompute picks up the shift within 7 days. No real-time update in M3.

## 4. Non-Functional Requirements

- **Performance:** Personalized recommendations p95 < 300ms (cached, precomputed); trending p95 < 200ms; similar p95 < 200ms (heavily cached, 1-hour TTL).
- **Availability:** 99.9%. If the recommendation service is unavailable, the client falls back to trending (which is a public, CDN-cached endpoint).
- **Scalability:** Serve 500k daily active users; each user hits the personalized endpoint once per home-page load. Cache hit ratio target > 90%.
- **Accessibility:** Shelves are horizontal carousels with keyboard navigation; reason badges have `aria-label` ("Recommended because you watched X").
- **Localization:** Anime titles use `title_jp` for Japanese locale, `title` otherwise. Reason strings are externalized.
- **Security:** Personalized endpoints require authentication; trending and similar are public. No user data in public responses. Rate limit 20 recommendation requests per 60 seconds per user/IP.

## 5. User Stories

- As a **logged-in user**, I want to see anime tailored to my taste so that I discover new shows I will enjoy.
- As a **new user**, I want to see trending anime so that I have a starting point even without watch history.
- As a **logged-in user**, I want each recommendation to explain why it was chosen so that I can trust the suggestion.
- As a **returning user**, I want the recommendations to reflect my latest watches so that the list feels fresh.
- As a **catalog browser**, I want to see similar titles on a detail page so that I can explore related shows.

## 6. Acceptance Criteria

- [ ] `GET /api/v1/recommendations` returns up to 10 personalized recommendations for authenticated users.
- [ ] Unauthenticated request to `/api/v1/recommendations` returns `401`.
- [ ] User with no history receives trending items with `reason: "fallback"`.
- [ ] Each recommendation includes `reason` and `reason_detail` fields.
- [ ] `GET /api/v1/recommendations/trending` is accessible without authentication.
- [ ] Authenticated trending request applies genre-boost.
- [ ] `GET /api/v1/recommendations/similar/{animeId}` returns up to 10 similar anime.
- [ ] Unknown `animeId` on similar endpoint returns `404 ANIME_NOT_FOUND`.
- [ ] Anime in user's watchlist or completed list is excluded from personalized recommendations.
- [ ] Response includes `meta.reason_counts` histogram.
- [ ] Cache headers match the spec: personalized `private, max-age=300, stale-while-revalidate=600`; trending `public, max-age=300, stale-while-revalidate=600`; similar `public, max-age=3600, stale-while-revalidate=7200`.

## 10. Edge Cases

1. **New user (cold start):** No history. Personalized endpoint returns trending with `reason: "fallback"`. The UI renders the shelf as "Trending Instead."
2. **User with homogeneous taste:** 90% of watch history is a single genre. The system caps genre affinity at 0.6 weight and distributes the remaining 0.4 to studio overlap, popularity, and exploration.
3. **User with no history and no trending data:** Impossible if the catalog has any anime. If catalog is truly empty, return `200` with `data: []`.
4. **Anime deleted after recommendation is computed:** The client receives an ID that no longer exists. The client should handle `404` on anime detail navigation gracefully (redirect to home or show a "no longer available" message).
5. **Recommendation staleness:** Precomputed lists are regenerated weekly. A user who binge-watches a new genre may see irrelevant recs for up to 7 days. Acceptable in M3; real-time recompute cost is prohibitive.
6. **Similar anime for an anime with no genre/studio overlap:** Every anime has at least one genre and one studio. If not (data integrity issue), the endpoint returns top-5 popular anime of the same type with `reason: "trending"`.
7. **Trending recs exhibiting filter bubbles:** The trending endpoint applies genre-boost for authenticated users. The boost is capped (max +20% position lift) to avoid burying globally popular titles.
8. **User in a non-supported locale:** The system falls back to English titles. `title_jp` is always available for CJK locales.

## 13. Security Considerations

- Personalized endpoints require authentication; user-specific data (watch history, bookmarks) is never exposed in public responses.
- Trending and similar endpoints are public and may be cached by CDNs. No PII in the response.
- Rate limit of 20 requests per 60 seconds per user/IP (shared across all four endpoints) prevents abuse.
- Precomputed recommendations are stored per-user. The key schema is `nexus:recs:user:{userId}:personalized`. A user may only read their own precomputed list; the key is derived from the session.
- Cache invalidation on watchlist/watch-progress change prevents stale recommendations from being served after a user's taste shifts.
- Recommendation responses are `Cache-Control: private` for personalized and `public` for trending/similar. No confusion between the two.
- Logging excludes user-specific recommendation scores and reason details to minimize PII in logs.

## 14. Performance Requirements

- **Personalized endpoint:** p95 < 300ms (served from precomputed cache; cache miss triggers real-time compute up to 500ms).
- **Trending endpoint:** p95 < 200ms (indexed query on `anime.popularity_score` + optional genre-boost).
- **Similar endpoint:** p95 < 200ms (indexed join on `anime_genres` + `anime_studios`; 1-hour CDN cache).
- **Continue endpoint:** p95 < 150ms (already computed from `continue_watching` table).
- **Rendering strategy:** SSR for the home-page recommendation rail (first paint); CSR for tap-to-navigate and infinite scroll.
- **Client JS budget:** Recommendation rail adds < 6 KB gzipped to the home-page bundle.
- **Cache hit ratio:** Target > 90% for personalized recs (precomputed).
- **Precomputation schedule:** Weekly background job. Each user's recommendation list is stored in `@nexus/cache` under `nexus:recs:user:{userId}:v1`.
- **Fallback:** If the personalized recommendation service times out, the client renders the trending shelf instead.

## 15. Future Improvements

1. **Collaborative filtering** — add signal from similar users' Watch histories to improve recommendation relevance (M6).
2. **Real-time personalization** — recompute recommendations on every watch-event rather than weekly (M6).
3. **Scribbles** — allow users to explicitly mark "Not Interested" to tune recs in real-time (M5).
4. **A/B test framework** — serve different algorithm variants to compare engagement (M5).
5. **Seasonal and event-based recs** — boost anime relevant to current events, holidays, or upcoming seasons (M4).
6. **Recommended playlists** — curated sequences of anime with thematic continuity (e.g., "MAPPA marathon") (M7).
